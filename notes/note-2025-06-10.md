
# 시작 3

다시 같은 LLM 쓰레드에 이어서 다음과 같은 질문을 했습니다.

```어느정도 정해진 느낌도 있지만 이 앱을 3단계 이터레이션에 걸쳐 구현한다면 어떻게 단계를 쪼개야 할지 제안해 줘.```

그 결과는 다음과 같습니다.

- 전체적인 흐름과 이터레이션의 핵심
    - 최소 기능 시작 → 피드백 확인 → 기능 확장 → 시스템화
    - 각 단계마다 목표, 핵심 기능, 기술 스택, 성공 지표, 리스크 및 대응을 확인.
    - 핵심 가치를 잘 전달하는지 지속적으로 확인: 묘수 풀이를 통한 학습 효과

- 1단계: MVP(최소기능제품): 핵심 가치 검증 (2-3주)
    - 목표:
        - 서비스의 핵심 가치인 "문제풀이를 통한 학습 효과" 검증
        - 하루 5분 학습이 실제 경기 의사결정에 도움을 주는지 파악
        - 실전 문제 10개-15개로 "풀이 경험" 자체가 유효한지 검증
        - 문제 제작 피로도 확인. 소수의 양질 콘텐츠 생산에 집중.
        - 테스트 유저 5명-10명에게 배포
    - 구현 사항:
        - 정적 웹 페이지로 10개-15개 엄선된 문제 제공
        - 배포는 AWS S3 + CloudFront
        - tft.kiok.app 도메인 사용
        - CSV 기반 문제 데이터 구조 설계 (상황 이미지, 선택지, 정답, 해설)
        - 기본적인 문제 풀이 인터페이스 (단일 페이지 애플리케이션)
        - 정답 공개 후 간단한 해설 표시
        - 간단한 진행률 저장 (LocalStorage: 로그인 불필요)
        - Discord 링크를 통한 피드백 수집
        - Google Analytics
    - 검증 지표:
        - 초기 사용자의 문제 완주율(≥30 %)
        - Discord를 통한 정성적 피드백
        - 문제의 교육적 효과에 대한 사용자 반응
        - 방문-재방문 비율(≥30 %), 문제당 평균 체류 시간(≥30 초)
        - Discord 초대 클릭률(커뮤니티 진입 의향 지표, ≥30 %)

- 2단계: 문제 출제 규모화, DB에서 문제 콘텐츠 다루기, 커뮤니티 연결 (4-6주)
    - 목표:
        - 콘텐츠 규모화(50문제 - 100문제)
        - 운영자/고급 유저가 *직접 문제를 등록* 할 수 있는 파이프라인 마련 (구글 폼 등)
        - 지속적인 사용을 유도하는 기능 추가 (오늘의 문제?)
        - 토론으로 깊이 강화 (여전히 디스코드)
    - 구현 사항:
        - Supabase 기반 백엔드 구축
        - 문제 등록/검수 워크플로우: Google Form → Sheets → 웹훅 → Supabase 테이블 동기화 (스크립트)
        - 로그인: 문제 출제자에 대해서 제한적으로 오픈
        - 자동화 스크립트: GitHub Actions
        - 프론트엔드: 기존 코드 → Supabase API 호출 방식으로 리팩터
        - 간단한 진도 추적 및 기본적인 성과 통계 기능 (로컬 스토리지 활용, 연속 참여일, 푼 문제 수)
    - 검증 지표:
        - 문제 출제자 수 ≥ 5명/월: 커뮤니티 기반 콘텐츠 공급 검증
        - 7일 리텐션 ≥ 20 %
        - 토론 참여자 수 ≥ 5명/월
    - 리스크/대응
        - 권한 남용 → 검수 대기열/버전 관리 필드로 롤백 용이하게 설계
        - 데이터 동기화 지연 → 재시도 로직 + 에러 웹훅 알림

## 대안적 관점

귀하의 계획에는 명확한 강점이 있으나, 몇 가지 다른 관점도 고려해볼 필요가 있습니다:

1. 콘텐츠 의존성: 이 서비스는 양질의 문제에 전적으로 의존합니다. 1단계에서 10문제로 시작하는 것은 검증에는 충분하나, 2단계로 넘어가기 전에 최소 50-100문제는 확보해야 사용자 이탈을 방지할 수 있을 것으로 예상됩니다.

2. 타겟 세분화: "롤토체스 실력을 높이고 싶은 사람"이라는 타겟은 다소 광범위합니다. 티어별(골드-플래티넘 구간 등) 또는 플레이 스타일별로 초기 타겟을 더욱 세분화하면 더 효과적인 문제 설계가 가능할 것입니다.

3. 수익화 시점: 3단계에서 수익 모델을 실험하는 것은 적절하나, 2단계부터 선택적 후원(Buy me a coffee 형태) 같은 가벼운 수익화를 시도해보는 것도 고려해볼 만합니다.

이러한 단계별 접근은 리스크를 최소화하면서도 핵심 가치를 빠르게 검증할 수 있는 전략으로 판단됩니다.

## 3단계 — 


핵심 기능

1. Riot API + 시즌 데이터 파이프라인 (사실)
    * 신규 패치 → ETL 작업(챔피언·아이템·시너지) → 문제 생성 매개변수 업데이트
2. 문제 자동 생성(추정)
    * 파라미터 기반 시나리오 샘플러 + LLM 해설 초안 생성
    * 운영자가 ✔️ 검수 → 실시간 배포
3. 스페이스드 리피티션 알고리즘
    * 알고리즘 자체는 검증된 SM-2 변형(사실), 챔피언·시너지 태그별 난이도 조절(추정)
4. 모바일 앱 (React Native Expo 또는 Flutter)
5. 수익화 실험
    * 광고 + 프리미엄 구독(무광고·추가 통계·고급 해설 동영상) A/B 테스트
    * e-스포츠 팀·스트리머 협찬 문제 세트(브랜디드 콘텐츠)

성공 지표

* 월간 활성 사용자(MAU) ≥ 10 k
* 구독 전환율 ≥ 2 %
* 자동 생성 문제 / 총 문제 비중 ≥ 60 % → 콘텐츠 확장성 확보

리스크·대응

* *자동 생성 품질 저하* → Human-in-the-loop·힌트 투표·신뢰도 점수
* *API Rate-Limit* → 캐싱&비동기 Job Queue, 필요한 범위만 부분 업데이트
* *운영 비용* → 서버리스+Edge First 구조로 트래픽 스파이크 완충

### 대안 시나리오 제시 (다른 관점)

* 초기부터 Notion DB + Super.so : 콘텐츠·페이지 관리 UI가 직관적이며 기여자 온보딩이 빠름. 단, 쿼리·API 자유도가 낮아 2단계 이후 확장성이 떨어진다.
* Firebase 전면 도입 : Authentication·Realtime DB·Cloud Functions 일체형. 리스너 기반 UI는 편리하지만 SQL형 분석·백업이 번거롭고 가격 체계가 복잡하다.
* “퍼즐 번들” 유료 PDF/App : 문제 제작 비용을 먼저 회수하되, 인터랙티브 경험이 줄어들어 핵심 차별화(인출 연습·간격 반복)가 약화될 수 있다.

* 1단계에서 얻은 실사용·설문 데이터가 이후 모든 의사결정의 기준이 됩니다.

무엇보다, 초기 반응이 미미하더라도 그것은 제품 방향을 조정할 기회임을 잊지 마십시오. 시행착오를 겪더라도 학습 곡선을 사용자와 함께 그린다는 점에서 *결국 서비스 자체가 하나의 실전 문제 풀이 과정*이 될 것입니다. 응원합니다.
