
# 시작 2

같은 LLM 쓰레드에 이어서 다음과 같은 질문을 했습니다.

```이 앱을 구현하는 방법을 제안해 줘. 기존에 존재하는 솔루션의 조합도 적극적으로 제시해 줘. 예를 들어 정보를 입력받아야 하면, 백엔드를 직접 구현하는 대신 구글 폼 + 스프레드시트 조합을 쓴다거나 하는 식으로.```

그 결과는 다음과 같습니다.

- 문제 데이터 입력/관리
    - 구글 폼을 사용해 문제를 입력
    - 구글 스프레드시트에 저장
    - 제출된 문제는 수동 검수 후 추가 (이 과정도 스프레드시트 기반)
    - 구글 스프레드시트 API를 이용해 앱에서 바로 문제를 불러옴
    - 나중에 성장하면 별도 데이터베이스 (예: Firebase, Supabase 등)로 이전
    - 나중에 앱 내에서 문제 출제를 받는 기능으로 확장

- TFT 게임 정보 업데이트
    - 구글 스프레드시트에 최신 게임 정보를 정리
    - 나중에 Riot API 직접 연동해서 챔피언/아이템/시너지 정보 획득

- 문제풀이 (프론트엔드)
    - 노코드 앱 빌더 사용 (예: Glide, Adalo, Thunkable)
    - 특히 Glide는 구글 스프레드시트 기반 앱을 바로 만들 수 있어서 이 케이스에 적합.

- 사용자 인증/기록
    - 구글 로그인 연동 (노코드 빌더에서 지원)
    - 또는 아예 인증 없이 시작할 수도 있음 (빠른 MVP 출시를 위해)
    - 개인 기록 (예: 문제풀이 수, 정확도)은 간단히 스프레드시트에 저장하거나, 후에 Firebase Authentication + Firestore로 확장 가능
    - 개인 맞춤 빌드 추천용 사용자 데이터 입력
    - 구글 폼으로 간단한 설문 제작하고 결과를 스프레드시트에 저장

- 커뮤니티 기능
    - Discord 서버를 만들어 초대
    - 앱 안에서 Discord로 링크 연결
    - 문제 해설 토론, 시즌별 메타 토론 등을 디스코드 내에서 진행

- 추가로 생각할 점
    - 초기 런칭 목표를 너무 높게 잡지 마세요. "10문제만 풀 수 있어도 출시"를 목표로 하세요.
    - MVP 핵심 가치는 문제풀이 경험 그 자체입니다. 부가기능은 이후에.
    - 초기 반응을 보고 나서 필요하면 점진적으로 직접 백엔드를 구축해도 늦지 않습니다.

---

문제 관리를 구글 시트를 사용한다는 점은 좋다고 생각했습니다. 하지만 아이디어 검증을 위해서는 구글API까지 갈 필요 없이 시트에 있는 CSV파일 웹 게시 정도로 충분하다고 생각했습니다. 정교한 데이터 필터를 위해서 백엔드와 데이터베이스가 필요해지기는 하겠지만 나중에 구글 시트에서 읽어가서 데이터베이스에 동기화 시키는 스크립트를 만드는 방식도 나쁘지 않겠다고 생각했습니다.

게임정보 업데이트는 결국 API 연동이나 대규모 적용 스크립트가 필요할 것이지만 지금 단계에서는 생각하지 않기로 했습니다.

프론트엔드는 노코드나 로우코드보다 바이브 코딩으로 처리해보려고 했습니다.

사용자 인증/기록은 첫 빌드 목표에서는 빼고 로그인이 필요해질 때 Supabase에서 처리하는게 좋겠다고 생각했습니다. 그리고 문제 관리 역시 나중에는 Supabase에서 서비스하면 좋겠지요.

커뮤니티는 Discord! 끝!

10문제를 출제해 본다는 목표는 아주 좋다고 생각했습니다.

그래서 첫 목표는,
- 구글 시트로 문제 관리. 프론트엔드에서 직접 CSV 읽어가기.
- 프론트엔드는 바이브 코딩으로, 바이브 코딩에 유리할 것으로 생각되는 React 기반의 Next.js + Tailwind로 작성.
- 커뮤니티로 Discord
- 구글 시트에서 바로 10문제 만들어 보기

나중에,
- Supabase로 백엔드 + 데이터베이스
- 구글 시트와 Supabase 싱크하는 스크립트 작성
- 문제 입력용 구글 폼

더 나중에,
- Supabase로 사용자 로그인
- Riot API 등 연동

다음 포스트에서는 LLM에게 3단계 이터레이션으로 나누어 구체적인 목표를 잡아보겠습니다.
